# Post-build patcher — зачем он нужен

`scripts/pipeline/postbuild_patch.py` — Python-скрипт, который правит BSL-файлы переведённого проекта **после** сборки зависимого перевода в EDT. Он существует потому что у EDT LanguageTool есть несколько особенностей, которые **нельзя починить одним только словарём**.

## TL;DR

Перевод в EDT применяет пословарный лукап к **каждому вхождению токена**, включая:

- переименование идентификаторов (нужно);
- имена регионов (нужно);
- **содержимое строковых литералов** (не нужно);
- **строковые литералы, которые должны остаться в исходном языке** (HTTP-глаголы, format-директивы платформы, byte-level тестовые данные и т.п.).

Словарь **не видит контекста**: одна и та же запись управляет и «переведи идентификатор `Post` как `Post`», и «переведи литерал `"POST"` как `"Post"`» — требования противоположные. `postbuild_patch.py` чинит последствия там, где словарь бессилен.

## Режим #1 — у EDT неверный английский алиас для некоторых платформенных встроенных функций

### Что ломается

EDT содержит встроенный словарь «platform context», который мапит русские имена платформенных идентификаторов на английские алиасы. Для **некоторых** встроенных он ошибочен:

| Русский | EDT мапит (неверно) | Реальное имя в 1С |
|---------|---------------------|-------------------|
| `СтрНачинаетсяС` | `StrStartsWith` | `StrStartWith` (без «s» в «Start») |

Compile-time-проверка EDT принимает ошибочное имя (доверяет своему маппингу). В **runtime** неверная форма падает с:

```
Procedure or function with the specified name is not defined (StrStartsWith)
```

Заметка: `СтрЗаканчиваетсяНа` → `StrEndsWith` ПРАВИЛЬНО (с «s»). Паттерн несимметричный — не применяйте преобразование «убрать s» к обоим случаям вслепую.

### Почему один словарь это не чинит

Мы добавляем `СтрНачинаетсяС=StrStartWith` в `common-camelcase_en.dict` (`fix_str_builtins.py`). Пользовательский словарь приоритетнее platform context, поэтому EDT использует правильное имя. **Это и есть основной фикс.**

Но:
- Промежуточные сборки (до подхвата записи) дают неверный вывод.
- Если EDT регенерит файл словаря, override может пропасть.
- Если разработчик добавит новый код с `СтрНачинаетсяС` без регенерации словаря, снова подхватится дефолтный platform.

Поэтому patcher держит страховку: заменяет `StrStartsWith(` → `StrStartWith(` на call-сайтах. Матчится с хвостовым `(` чтобы не трогать комментарии.

## Режим #2 — camelcase-словарь переводит содержимое строковых литералов

### Что ломается

Camelcase-словарь EDT предназначен для перевода **BSL-идентификаторов** (CamelCase-композитов). Но EDT также **токенизирует содержимое строковых литералов** тем же словарём и заменяет совпавшие токены.

Пример — русский тест:

```bsl
Ключ = ПолучитьДвоичныеДанныеИзСтроки("Секретный ключ", КодировкаТекста.UTF8, Ложь);
Данные = ПолучитьДвоичныеДанныеИзСтроки("Какие-то данные", КодировкаТекста.UTF8, Ложь);
```

В словаре есть токены типа `Секретный=Secret`, `ключ=key`, `Какие=Which`, `то=then`, `данные=data`. После перевода EDT:

```bsl
Var_Key = GetBinaryDataFromString("Secret key", TextEncoding.utf8, False);
Data = GetBinaryDataFromString("Which-then data", TextEncoding.utf8, False);
```

Для тестов, которые сравнивают **байты** (HMAC, хеш, URL-encoded содержимое), это **меняет байты**, подаваемые в хеш-функцию, и ломает assertion.

### Почему один словарь это не чинит

Проблемные токены — легитимные части **настоящих идентификаторов**. Удаление токен-уровневых записей ломает перевод любого композита, у которого нет явного full-identifier override. EDT не даёт ограничить область словаря только идентификаторами.

### Как patcher это чинит

`restore_literals()` в phase 2:

1. Построчный diff переведённого файла с русским исходником.
2. Для каждой строки со строковым литералом, отличающимся в RU и EN версиях:
   - если в RU-литерале нет кириллицы → оставить переведённым;
   - если RU-литерал «identifier-like» (`[A-Za-zА-Яа-яЁё0-9_]+`, опционально `,` + `\s`) → оставить переведённым (имена процедур для `Execute()`, списки ключей структур);
   - иначе → вернуть EN-литерал к русскому оригиналу.

Настройка: `LITERAL_RESTORE_PAIRS` — список `(ru_path, en_path)` для каждого модуля, где есть тестовые данные в литералах. Для УИ_ заполнять по факту падения тестов.

## Режим #3 — HTTP-глаголы / другие casee-sensitive литералы case-folding'ятся

### Что ломается

Тот же механизм токенизации, но задетые литералы — **identifier-like**, поэтому эвристика literal-restore их не ловит.

Русский исходник:

```bsl
Возврат ВызватьHTTPМетод(ТекущаяСессия, "POST", URL, Параметры);
```

В словаре есть `Post=Post` (PascalCase — правильно для BSL-имени функции). EDT case-insensitive совпадает литерал `"POST"` с токеном `Post` и выдаёт значение словаря `Post`:

```bsl
Return CallHTTPMethod(CurrentSession, "Post", URL, Parameters);
```

`"Post"` передаётся как HTTP-метод. HTTP-серверы case-sensitive по спецификации (RFC 7230); сервер возвращает `400 Bad Request`.

### Как patcher это чинит

Явные контекстно-привязанные замены, скопированные по call-сайтам. Префикс сужает матч (`CallHTTPMethod(CurrentSession, ` или другой), чтобы произвольные `"Post"` в других местах файла не трогались. У УИ_ есть общий модуль `УИ_КоннекторHTTP` — если по нему пойдут проблемы, шаблон тот же что в HTTPConnector.

## Режим #4 — поля платформенных объектов с обратным порядком слов

### Что ломается

`common-camelcase_en.dict` строит перевод композитов конкатенацией переводов токенов **в том же порядке** что русские. У многих свойств/методов платформенных объектов порядок слов другой:

| Русский | Позиционный перевод (неверно) | Реальное имя в платформе |
|---------|-------------------------------|---------------------------|
| `КодСостояния` | `CodeStatus` | `StatusCode` |
| `ПереносСтрок` | `BreakLines` | `NewLines` |
| `СимволыОтступа` | `SymbolsIndent` | `PaddingSymbols` |
| `ЭкранироватьРазделителиСтрок` | `EscapeSeparatorsLines` | `EscapeLineTerminators` |
| `АдресРесурса` | `AddressResource` | `ResourceAddress` |
| `ИспользоватьАутентификациюОС` | `UseAuthenticationOS` | `UseOSAuthentication` |

Когда переведённый модуль обращается к полю по неверному имени:

```
Object field not found (CodeStatus)
```

### Почему один словарь это не чинит (полностью)

В принципе чинит — явным full-identifier override (`КодСостояния=StatusCode`) через `fix_platform_fields.py`. Но:

- Несовпадения обнаруживаются реактивно — когда тест падает с «Object field not found».
- Если разработчик добавит новый код с не встречавшимся ранее композитом — поломается молча до первого тестирования.

Patcher держит параллельный список замен на access-сайтах (`.CodeStatus` → `.StatusCode`, `"CodeStatus"` → `"StatusCode"`) как независимую страховку. Фикс в словаре — основной; patcher — подстраховка.

## Почему две фазы?

`postbuild_patch.py` делит работу на две фазы:

- **Фаза 1 — замены идентификаторов/литералов.** Простые замены пар строк. Детерминированные, быстрые, идемпотентные. Покрывают режимы #1, #3, #4.
- **Фаза 2 — восстановление литералов из RU-исходника.** Для модулей, где RU-исходник — авторитет по содержимому строковых литералов (тестовые модули), делает построчный diff и возвращает non-identifier-like русские литералы на место. Покрывает режим #2.

Фазы независимы и композируются по каждому файлу.

## Когда расширять

- **Новая runtime-ошибка `not defined` для платформенной встроенной функции** — добавить в `REPLACEMENTS` режима #1 И в overrides `fix_str_builtins.py`.
- **Новый падающий тест на byte-уровне** — зарегистрировать пару `(ru_path, en_path)` модуля в `LITERAL_RESTORE_PAIRS`.
- **Новая ошибка `Object field not found`** — подтвердить реальное имя свойства через `get_platform_documentation` MCP-tool или 1С-доки, добавить в overrides `fix_platform_fields.py` И в `REPLACEMENTS` режима #4.
- **Новый тест падает с `400 Bad Request` от сервера на HTTP-вызове** — вероятно, ещё один HTTP-глагол / протокольный токен покоцан case-fold'ом; добавить контекстно-привязанную пару в HTTP-verb-блок (паттерн как в HTTPConnector).

Каждое расширение по одному паттерну: выяснить правильную английскую форму (через `get_platform_documentation` или 1С-доки), добавить явный override в словарь где можно, и safety-net замену в patcher.

## Стартовая поставка для УИ_

В отличие от HTTPConnector, для УИ_ patcher стартует пустым по части project-specific фиксов — таблица `REPLACEMENTS` содержит только generic 1C-platform правила (StrStartWith + 6 платформенных полей). Project-specific записи (HTTP-verbs, literal restore, ещё какие-то режимы) добавляются по факту обнаружения после первой полноценной сборки и тестов.
